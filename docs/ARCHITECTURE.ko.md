# 구조

> [English](ARCHITECTURE.md)

```text
Codex 로컬 상태(~/.codex) ───────────────┐
Codex 렌더러(임의 127.0.0.1 CDP) ────────┼─ src/plugin.js ── 로컬 WebSocket ── Stream Deck
Codex 손쉬운 사용 메타데이터 ────────────┤          │
CodexBar CLI(사용량만) ──────────────────┘          ├─ control-plane.js
                                                       ├─ micro-cdp.js(우선)
                                                       ├─ src/*.js 도메인 모듈
                                                       └─ keybridge(기존 방식+macOS 오디오·AX)
```

ThreadDeck 런타임은 외부 의존성 없이 동작합니다. Node.js 프로세스가 작은 CommonJS 도메인 모듈, 시각 렌더링, Stream Deck 이벤트, 선택적인 Codex Micro 렌더러 어댑터를 조합합니다. macOS 입력·오디오·손쉬운 사용·호환 동작은 범용 네이티브 헬퍼가 유지합니다.

제어부는 Micro 우선이지만 Micro 전용은 아닙니다. Micro 이벤트가 전달되기 전 `사용 불가`가 확정된 경우에만 기존 어댑터로 폴백합니다. 전달 뒤 타임아웃·연결 종료·검증 실패는 모호한 결과로 취급해 손쉬운 사용 경로로 재실행하지 않습니다. 따라서 Fast가 두 번 토글되거나 메시지·작업 전환이 중복 실행되지 않습니다.

## 런타임 구성 요소

### Node.js 플러그인

`src/plugin.js`는 조립 지점이자 런타임 진입점입니다. 외부 I/O, 캐시, 타이머, SVG 렌더링, Stream Deck WebSocket, 네이티브 헬퍼 호출을 담당합니다. 빌드는 최상위 `src/*.js` 모듈을 각각 대응하는 `com.yechan.threaddeck.sdPlugin/bin/*.js` 경로로 바이트 단위 변형 없이 복사합니다. manifest는 `bin/plugin.js`를 시작하고, 이 파일이 같은 디렉터리의 CommonJS 모듈을 불러옵니다.

주요 역할은 다음과 같습니다.

- 로컬호스트 플러그인 WebSocket으로 Stream Deck에 연결
- 로컬 SQLite의 고정·최근 작업과 Codex JSON 상태에서 사용자가 직접 고정한 원격 작업만 대화 ID 기준으로 병합
- 작업 중·완료·대기·오류·유휴 상태 정규화
- 원격 시작은 UUIDv7 턴 ID에서 복원하고 종료는 명시적 수명 주기 이벤트 또는 확인된 실시간 상태 전환에서만 고정하며, 요약 갱신 시각은 종료값으로 사용하지 않음
- 원격 reasoning-summary 원문은 노출하지 않고 별도의 턴 범위·단조 시각 활동 캐시에서 보수적인 중간 단계로만 정규화
- 현재 Desktop 세션을 추적해 닫힌 임시 사이드챗이 일반 작업처럼 다시 나타나는 현상 방지
- ThreadDeck 소유 144 × 144 버튼 전체를 SVG로 렌더링
- 작업 중 추론 단서, 대기열 진행 알림, 완료 펄스 애니메이션
- 선택적인 주간 한도 값 확인을 위해 CodexBar 호출
- 키보드·미디어·음성 입력 동작을 `keybridge`에 위임
- 음성 입력 미디어 전환을 소유권 lease 뒤에서 직렬화해 여러 음성 버튼이 겹쳐도 한 번만 멈추고 마지막 소유자만 재생을 재개하도록 관리
- 작업 버튼을 누르고 있으면 해당 작업을 열고 후속 요청을 받아쓴 뒤 텍스트 지문으로 완료를 확인해 버튼을 놓은 후 제출
- 녹음 시도마다 단조 증가하는 세션 토큰을 부여하고, 어느 작업 버튼에서든 새 hold가 시작되면 이전 전역 입력창 세션을 대체하며, 비동기 받아쓰기·제출 콜백이 현재 세션을 바꾸기 전에 같은 토큰인지 확인
- 원격 작업 열기를 대상별 단일 실행으로 관리해 같은 작업의 반복 입력은 합치고 다른 새 대상은 오래된 전환을 취소하며, 손쉬운 사용 컨트롤에서 정확한 UUID를 우선하고 통합 검색은 최대 한 번만 연 뒤 가벼운 포커스 헤더 검사로 Codex가 올바른 작업과 컴퓨터를 활성화했는지 확인
- 사용자 제어를 `control-plane.js`로 보내 Codex Micro를 우선 사용하고 Micro 이벤트가 전달되지 않았다고 확정된 경우에만 기존 네이티브 어댑터 사용
- 독립된 8개 작업 모니터를 유지하면서 Codex Micro 슬롯 6개에서 정확한 식별자가 확인된 작업은 `AG00`~`AG05` 직접 전환에 연결

Codex 파일은 쓰기 모드로 열지 않습니다.

### 네이티브 헬퍼

`native/keybridge.m`은 `scripts/build-bridge.sh`에서 `arm64`와 `x86_64`로 각각 컴파일한 뒤 하나의 범용 실행 파일로 합칩니다.

주요 역할은 다음과 같습니다.

- Codex 단축키와 Return 입력
- Stream Deck 버튼을 누르고 있는 동안 음성 입력 보조키를 홀드하고 놓을 때 해제
- 물리 D 키 코드를 유지하면서 라틴 `D` 문자를 음성 입력 이벤트에 명시해 현재 키보드 입력 소스와 무관하게 단축키 실행
- 앱 전환과 미디어 키 입력
- 앱 허용 목록 없이 활성 Core Audio 프로세스마다 화면 앱 소유자를 찾고 의미가 확인된 일시정지·재생 컨트롤 확인
- 해당 컨트롤을 직접 사용하고 필요할 때만 macOS 정상 미디어 명령으로 폴백하며, PID나 미디어 문구 대신 번들 식별자만 10분짜리 로컬 임대 정보에 기록해 놓을 때 정확한 앱만 재개
- 화면에 보이는 Codex 손쉬운 사용 트리를 대상 UUID와 정규화한 제목 지문 전체로 한 번 순회하고, 식별이 안전한 누를 수 있는 결과만 활성화하며, 엄격 모드에서는 제목만 같은 중복을 거부하고, 안전 동작용 전면 작업과 표시용 Codex 내부 활성 창 작업을 각각 헤더에서 확인하며, 메시지 원문을 반환하지 않고 대기 동작 개수를 계산

헬퍼는 macOS 시스템 프레임워크만 사용합니다. 합성 입력에는 Stream Deck 손쉬운 사용 권한이 필요합니다. 렌더러 연결 뒤에는 Micro 기반 Effort·Fast·사이드챗·일반 보내기·새 작업·PTT·6개 슬롯 전환이 합성 단축키를 요구하지 않습니다. 헬퍼는 미디어 처리, 작성창 검증, 대기열, 원격·비슬롯 전환, 긴 보내기, 호환 폴백에서 계속 기준 역할을 합니다.

### Codex Micro 렌더러 어댑터

`src/micro-cdp.js`는 `127.0.0.1`에 명시적으로 바인딩된 디버거만 허용하고 `/json/version`과 메인 `app://` 렌더러를 모두 검증한 뒤 루프백 CDP WebSocket으로 통신합니다. 주기 스냅샷은 상태를 바꾸지 않으며 현재 작업 ID, 다음 응답 Effort, Fast, 테마, 개인정보를 제한한 슬롯 6개, 기능 불리언만 반환합니다. 프롬프트·응답·대기 메시지·받아쓰기 원문은 반환하지 않습니다.

Micro 동작 활성화는 지연 실행됩니다. HID/PTT가 필요한 첫 물리 입력에서만 현재 렌더러 메모리에 기능 게이트 override를 설치하고 Micro 연결 이벤트를 보냅니다. 공식 keycap 명령은 Fast·사이드챗·보내기·새 작업을, host PTT 이벤트는 마이크를 담당합니다. Effort는 Codex가 `ENC_CW`/`ENC_CC`와 `act: 2`에서 호출하는 것과 같은 내부 `composer.openModelPicker` power-selection 명령을 사용하고, `AG00`~`AG05`는 슬롯 전환을 담당합니다. override는 해당 렌더러 프로세스가 끝나면 사라집니다.

`src/micro-bootstrap.js`는 ThreadDeck이 처음 관찰했을 때 이미 실행 중인 Codex 세대를 중단하지 않습니다. 사용자가 나중에 Codex를 정상 종료·실행한 뒤에도 연결이 없는 상태가 안정적으로 확인되면 임의 루프백 포트로 한 번만 보호된 재실행을 수행합니다. 시도는 프로세스 세대별로 한 번이고 10분 간격 제한이 있으며 `THREADDECK_DISABLE_MICRO_BOOTSTRAP=1`로 자동 복구를 끌 수 있습니다. `~/Library/Application Support/ThreadDeck`에는 프로세스 세대·포트·상태·간격·숫자 시각만 저장합니다.

### Neo 프로필

편집 가능한 프로필 원본은 `profiles/source/unpacked`에 있습니다. `scripts/build-profile.sh`가 플러그인 manifest에서 참조하는 `.streamDeckProfile` 압축 파일을 만듭니다. 공개 전에 하드웨어 UUID를 제거하므로 가져온 프로필이 제작자 기기에 묶이지 않습니다.

ThreadDeck은 기본 프로필의 이전 페이지 액션을 소유하고 다음 페이지 액션도 제공합니다. Stream Deck의 공식 `switchToProfile` 명령에 페이지 번호를 보내므로 탐색은 기본 방식으로 작동하면서 완료 효과에도 참여합니다. Elgato 소유 앱 실행 액션은 기본 렌더링을 유지하므로 ThreadDeck 오버레이가 적용되지 않습니다.

대시보드 키패드 `0,1`에는 현재 작업, `1,1`에는 추론 강도/Fast 결합 버튼을 둡니다. 새 작업·사이드챗·보내기·마이크·사용량·뒤로가기 버튼이 나머지 칸을 채웁니다. Fast 전용 액션과 앱 전환 액션도 사용자 지정 프로필을 위해 액션 목록에 계속 제공합니다.

### 언어와 플랫폼 경계

Stream Deck은 등록 정보에 `application.language`와 `application.platform`을 전달합니다. ThreadDeck은 시작할 때 언어를 한 번 정규화하고 지원하지 않는 언어는 영어로 폴백하며, 안정적인 도메인 활동 코드를 `src/i18n.js`에서 화면 문구로 바꿉니다. 같은 패키지가 액션 목록용 `en.json`과 `ko.json`도 포함하므로 영어판·한국어판 바이너리를 따로 만들지 않습니다.

manifest는 현재 macOS만 선언합니다. `src/runtime-info.js`가 플랫폼 기능 경계를 제공하고, 네이티브 입력·손쉬운 사용·오디오·활성 앱 동작은 모두 `keybridge` 뒤에 둡니다. 향후 Windows 이식은 렌더러와 수명주기 리듀서 곳곳에 분기를 추가하지 않고 동일한 검증 명령 계약을 구현하는 플랫폼 어댑터 방식이 적합합니다. 자세한 내용은 [플랫폼 이식](PORTING.ko.md)을 확인하세요.

## 데이터 갱신과 렌더링

- ThreadDeck 소유 액션이 하나라도 보이는 동안 작업 메타데이터를 3초마다 갱신합니다.
- SQLite 행, 원격 요약, 사이드챗은 선택 전에 하나의 개인정보 경계를 통과합니다. 사이드바 제목을 덮어쓰기 전에 서브에이전트 출처를 구조적으로 거부하고, 구버전 메타데이터에는 정확한 guardian·ambient 프롬프트 서명을 보조 필터로 사용합니다. 별도의 영구 ID 조회로 숨긴 서브에이전트가 프롬프트 기록을 통해 사이드챗으로 재분류되는 경로도 막습니다.
- 작업 갱신마다 Codex 전역 상태 파일을 한 번 읽고 같은 Promise를 고정 ID, 원격 요약, 사이드챗 프롬프트 기록 파서가 공유합니다. 따라서 세 파서가 같은 파일 세대를 관찰하며, 읽기에 실패하면 서로 다른 읽기의 필드를 섞지 않고 각 소비자가 기존의 안전한 대체 동작을 적용합니다. 사이드챗 탐색은 프롬프트 기록에 더해 Desktop 로그의 성공한 `thread/fork` → `thread/inject_items` 쌍을 증분 축약합니다. 따라서 프롬프트 기록이 형제 사이드챗 하나를 누락해도 정확한 부모와 서로 다른 임시 UUID를 모두 보존하고, 정확한 `thread/unsubscribe`가 기록된 UUID 하나만 닫습니다. 탐색 상태와 바이트 커서는 app-server 세션 경계에서 초기화하며 일시적인 상태 파일 재작성 중에는 마지막 검증 목록을 유지합니다.
- 고정한 로컬·원격 작업을 먼저 배치하고 남은 최근 슬롯은 로컬 작업과 사이드챗만 채웁니다. 같은 대화 ID가 양쪽에 있으면 로컬 레코드를 우선합니다.
- 대시보드가 보일 때 750ms 간격으로 읽기 전용 Micro 현재 ID와 기존 손쉬운 사용 창 검사를 합쳐 Codex에서 직접 바꾼 작업과 모든 종속 제어를 1초 안에 갱신합니다. 보내기·마이크·Effort·Fast·사이드챗은 실행 직전 같은 작업을 다시 확인합니다. 새 사이드챗 UUID 전에는 위치까지 확인된 오른쪽 작성창만 임시 대상으로 허용하고 발견 즉시 실제 UUID로 바꿉니다. 대기 컨트롤은 작업 영역별로 연결해 일반 작업과 사이드챗 개수가 독립적으로 감소합니다.
- 플러그인 시작과 Codex App Server·렌더러 세대 변경 뒤 `config.toml`과 `models_cache.json`에서 Effort 목록을 미리 읽고 Micro 작성창의 읽기 전용 값과 보정합니다. 클릭할 때마다 320ms 가감속 트랙 애니메이션을 즉시 시작하고 연타 중에도 Codex 적용을 기다리지 않고 계속 그립니다. 일반 Micro 도달 단계는 90ms 뒤 마지막 내부 노브 이동만 전달합니다. `Max`·`Ultra` 또는 노브가 요청 단계를 건너뛴 경우에는 입력이 1.1초 멈춘 뒤 정확한 `Advanced`를 열고 실제 계정·모델 옵션을 다시 스캔해 목표만 선택하며, 고정된 Ultra 전체 접근 경고만 처리한 다음 작성창 포커스를 복원합니다. 끝 단계에서는 방향을 반전합니다. 600ms 길게 누르면 Fast 내부 명령을 즉시 실행하고 놓을 때 다시 실행하지 않습니다.
- 원격 Desktop 로그는 파일별 바이트 위치와 제한된 raw-byte carry로 이어서 읽습니다. 완성된 UTF-8 줄만 상태에 반영하므로 긴 reasoning-summary 한 줄이나 다중 바이트 문자가 두 번의 폴링에 걸쳐 기록되어도 다음 폴링에서 완성됩니다. 파일 identity와 경계 지문으로 회전·truncate·같은 경로의 빠른 교체를 감지한 뒤에만 기존 커서를 재사용합니다.
- 작업 중 시간과 애니메이션은 실제 버튼에 맞는 주기로 렌더링합니다.
- ThreadDeck 소유 액션이 보이는 동안 주간 사용량을 60초마다 갱신해 한도 페이지를 열기 전에 값을 준비합니다.
- macOS 화면 모드를 2초마다 확인해 기존 다크·라이트 팔레트 사이를 전환합니다.
- 완료 감지는 시작 유예 시간과 겹치는 관찰 구간을 이용해 새 종료 시각을 비교합니다. 대기열 관찰은 오래된 전환 상태를 차단하지만 그 자체를 완료 근거로 사용하지 않습니다. 첫 전역 프레임은 화면에 보이는 모든 ThreadDeck 소유 버튼에 먼저 전송하고 이후 프레임만 기기 처리량에 맞춰 나눕니다. 해당 작업 버튼은 더 긴 첫 펄스 뒤에 전송량을 낮춘 호흡 효과를 유지하며, 전환 성공 또는 Codex 전면의 정확한 작업 일치가 확인되어야 로컬 미확인 상태를 해제합니다.

각 컨텍스트의 마지막 이미지를 캐시해 바뀌지 않은 프레임은 다시 보내지 않습니다.

## 빌드와 검증 경계

`scripts/build.sh`는 최상위 JavaScript 원본 모듈을 변환하지 않고 플러그인 `bin/` 디렉터리에 그대로 복제하며, 원본이 없는 과거 JavaScript 모듈은 제거합니다. `scripts/verify.sh`는 원본과 번들 복사본의 문법을 각각 확인하고 각 쌍을 바이트 단위로 비교합니다.

`pnpm run test`는 Node 기본 `node:test` 실행기로 `test/` 아래 I/O 없는 모듈 계약을 검증합니다. 런타임 진입점에는 컨텍스트와 캐시 조정이 필요한 완료 효과 전파, 갱신 복원력, 사용량 캐시, 음성 제출, 빠른 모드 렌더링, 원격 전환 단일 실행·검색 횟수, 오래된 홀드 차단, 직렬화된 미디어 소유권 동작의 작은 인라인 계약 모드가 남아 있습니다. 전체 `pnpm run check`는 두 테스트 그룹과 네이티브 헬퍼, manifest, 문서, 릴리스 검사를 함께 실행합니다.

## 문서 렌더러

`scripts/render-docs.mjs`는 같은 버튼 렌더링 함수를 개인정보 없는 데모 모드로 실행해 다크·라이트 4 × 2 개요와 정확한 개별 버튼 PNG를 만듭니다. `scripts/render-animation.mjs`는 추론 트랙 단계 이동, Fast mode 길게 누르기와 활성화, 작업 음성 입력, 대기열 진행, 전체 완료 효과를 순서대로 보여주는 결정론적 72프레임 개요를 만들고, 작업 버튼 음성 입력·전용 마이크·보내기 길게 누르기·앱 실행 안내 GIF도 렌더링합니다. 공용 `scripts/rasterize.mjs` 헬퍼는 문서·자산 빌드 전용 Sharp 의존성으로 SVG를 결정론적으로 래스터화하고, 저장소가 소유한 Swift·ImageIO 헬퍼는 GIF를 인코딩합니다. ThreadDeck 집중 GIF의 버튼 상태는 배포 렌더러 출력 그대로이며 바깥에 한·영 타임라인만 추가합니다. Elgato 소유 앱 실행 동작은 실제 그림을 복제하지 않고 중립 안내 버튼으로 표현합니다. Sharp와 문서 인코더는 런타임 플러그인에 포함되지 않습니다.

## 안정성 경계

Stream Deck 플러그인 프로토콜과 프로필 manifest는 공개 API지만 `~/.codex` 아래 Codex 파일은 그렇지 않습니다. 파서는 다음 원칙을 지킵니다.

- 없거나 이름이 바뀐 선택 필드를 허용
- 제목보다 안정적인 ID를 우선
- 모든 접근을 읽기 전용으로 유지
- 상태를 변경하지 않고 안전한 대체 화면 표시
- 비공개 형식 경계를 작업 읽기 함수 안에 격리

Codex App Server에는 실험적인 작업별 Effort 갱신 메서드가 있지만, Codex Desktop은 외부 연결 수단이 없는 전용 stdio 하위 프로세스를 사용합니다. 별도 서버는 Desktop에 이미 열린 작업과 동기화하지 못하므로 ThreadDeck은 그림자 서버를 띄우거나 세션 파일에 쓰지 않습니다. 현재 렌더러 연결은 명시적으로 비공개 호환 계층입니다. 기능 탐색·한 번만 실행하는 폴백·안정된 `control-plane.js` 인터페이스를 통해 향후 공식 전송 경로나 Windows 어댑터가 생겨도 Stream Deck 렌더러와 작업 리듀서를 바꾸지 않고 교체할 수 있습니다.
